# HW12 – временные ряды: temporal split, baseline-модели и GRU-прогноз

## 1. Кратко: что сделано

Использовался датасет `S12-hw-dataset.csv` с почасовыми данными за первое полугодие 2025 года. Прогнозировалась целевая переменная target (числовой ряд).

Данные были разбиты на train / validation / test по времени в пропорции 70% / 15% / 15% без перемешивания.

Сравнивались 4 подхода:

- B1 (naive-last) – прогноз равен последнему известному значению;
- B2 (moving-average) – скользящее среднее с окном 7;
- B3 (ridge-lag-features) – гребневая регрессия на лаговых, rolling- и календарных признаках;
- R1 (gru-forecast) – рекуррентная нейросеть GRU с окном 24 часа.

Лучшей моделью по валидационной MAE оказалась GRU (5.30), которая и была протестирована на тестовой выборке.

## 3. Данные и постановка задачи

- Датасет: `S12-hw-dataset.csv`
- Целевая колонка: `target`
- Частота временного ряда: почасовые наблюдения
- Горизонт прогноза: 1 шаг (следующий час)
- Размер датасета: 4320 наблюдений (с 1 января по 29 июня 2025)
- Диапазон дат: 2025-01-01 00:00:00 – 2025-06-29 23:00:00
- Есть ли дополнительные внешние признаки: нет, только временная метка и целевая переменная
- Комментарий: Ряд имеет слабый восходящий тренд, ярко выраженную суточную сезонность (днём выше, ночью ниже) и недельную сезонность (выходные ниже будней). Пропусков нет, но есть отдельные выбросы (например, 210.1 12 июня). Нестационарность проявляется в изменении дисперсии: во второй половине ряда амплитуда колебаний заметно выше, чем в начале.

## 4. Temporal split и признаки

### 4.1. Разбиение по времени

Разбиение выполнено хронологически без перемешивания:

Train: первые 70% данных (3024 наблюдения)
Validation: следующие 15% (648 наблюдений)
Test: последние 15% (648 наблюдений)
Почему `random split` некорректен: случайное перемешивание нарушило бы временную структуру ряда, позволило бы модели "увидеть" будущее при обучении и дало бы нереалистично оптимистичную оценку качества.

### 4.2. Признаки для baseline-моделей

- Лаги: 1, 7, 14 часов
- Rolling-признаки: скользящее среднее и стандартное отклонение с окном 7 (со сдвигом, чтобы избежать утечки)
- Календарные признаки: день недели, час дня, месяц
- Обработка пропусков: после построения лагов и rolling-признаков строки с NaN были удалены
- Масштабирование: использовался StandardScaler на train-части

## 5. Модели и эксперименты (B1, B2, B3, R1)

- B1 (naive-last): прогноз = значение target за предыдущий час. Простейший baseline.
- B2 (moving-average): прогноз = скользящее среднее за предыдущие 7 часов.
- B3 (ridge-lag-features): гребневая регрессия (alpha=1.0 по умолчанию) на признаках: лаги 1,7,14; rolling_mean_7; rolling_std_7; dayofweek; hour; month.
- R1 (gru-forecast): GRU-нейросеть:
  - window_size: 24 часа (сутки)
  - архитектура: 2 слоя GRU, hidden_size=64
  - batch_size: 64
  - эпохи: 50 (с ранней остановкой по валидационной MAE)
  - optimizer: Adam, learning_rate = 0.001
  - сохранение модели: лучшая по val_mae

- Основная метрика выбора: MAE (средняя абсолютная ошибка)
- Дополнительные метрики: RMSE, MAPE

Выводы по валидации и тесту в целом совпали: GRU и Ridge существенно лучше простых baseline.

## 7. Анализ

Наивный baseline (B1) оказался неожиданно сильным для почасового ряда с высокой автокорреляцией – ошибка всего 6.44, что подтверждает, что следующий час часто похож на предыдущий. Скользящее среднее (B2) показало худший результат (12.70), так как сглаживание "размывает" суточные пики и не успевает за резкими изменениями.

Ridge-регрессия на лаговых и календарных признаках (B3) улучшила результат до 6.36, что логично: модель научилась учитывать час дня (суточная сезонность) и день недели. Это подтверждает важность календарных признаков для такого ряда.

GRU (R1) с окном 24 часа смогла уловить суточные паттерны и дала лучший результат на валидации (5.30). Однако на тесте её MAE выросла до 6.41, что немного хуже Ridge на валидации. Возможные причины:
- тестовый период (конец июня) может иметь немного другую динамику;
- GRU могла немного переобучиться под валидацию;
- для такой задачи Ridge с ручными признаками может быть достаточно.

Тренд и сезонность хорошо видны в данных, и обе лучшие модели (Ridge и GRU) смогли их уловить. Наиболее показательные ошибки GRU приходятся на утренние и вечерние часы, когда значения быстро меняются.

В такой задаче утечки данных могли бы возникнуть при:

- использовании будущих значений для построения rolling-признаков;
- случайном перемешивании train/test;
- масштабировании всей выборки целиком.

Выбор по валидации, а не по тесту, принципиален, чтобы не подгонять модель под финальную оценку. Тест должен использоваться только один раз – это и было сделано.

## 8. Итоговый вывод

Для данной задачи в качестве базового подхода можно рекомендовать Ridge-регрессию с лаговыми и календарными признаками – она даёт стабильный результат, легко интерпретируется и не требует тонкой настройки. Простого наивного прогноза недостаточно для учёта суточной сезонности.

GRU может быть полезна, если ряд сложнее (например, есть долгосрочные зависимости, нелинейности или многомерные входные данные). В данном случае прирост качества относительно Ridge оказался небольшим и нестабильным на тесте.

Главный вывод по корректной валидации временных рядов: всегда разбивайте данные по времени, не перемешивайте, используйте тест только для финальной проверки. Иначе вы рискуете получить оптимистичные, но нереалистичные оценки качества.
